# v0.34.0 — Team Collaboration v2: 团队协作增强（角色权限 + 审计增强）

> 状态：**计划中**
> 依赖：v0.33.0, v0.30.0 (团队协作 v1)
> 关联：在 v0.30.0 Git 仓库同步基础上，增加企业级角色权限和审计能力

---

## 背景

### v0.30.0 回顾

v0.30.0 实现了团队协作 v1：
- Git 仓库作为配置同步中心
- 团队服务器/片段共享
- 基于团队密钥的加密
- 简单的 admin/member/readonly 角色（客户端强制）

### v0.34.0 目标

将团队协作从 "能用" 升级到 "企业级"：

1. **细粒度角色权限** — 按服务器组/操作类型控制访问
2. **审计增强** — 完整的操作审计链，可导出合规报告
3. **凭证隔离** — 团队成员无法导出共享凭证明文
4. **成员管理** — 邀请/移除成员，权限变更通知
5. **冲突解决 UI** — 可视化合并冲突而非自动覆盖

## 一、细粒度角色权限

### 1.1 权限模型

从 v0.30.0 的 3 级角色扩展为基于能力（capability）的模型：

```rust
/// 团队角色定义
#[derive(Debug, Clone, serde::Serialize, serde::Deserialize)]
pub struct TeamRole {
    pub name: String,              // "admin", "ops", "developer", "viewer"
    pub display_name: String,      // "管理员", "运维", "开发者", "只读"
    pub capabilities: Vec<Capability>,

/// 细粒度能力
#[derive(Debug, Clone, PartialEq, serde::Serialize, serde::Deserialize)]
pub enum Capability {
    // 服务器操作
    ServerConnect,             // 连接服务器
    ServerCreate,              // 创建共享服务器
    ServerEdit,                // 编辑服务器配置
    ServerDelete,              // 删除共享服务器
    ServerViewCredentials,     // 查看密码/密钥明文
    
    // 片段操作
    SnippetCreate,
    SnippetEdit,
    SnippetDelete,
    SnippetExecute,
    
    // 团队管理
    TeamInvite,                // 邀请新成员
    TeamRemove,                // 移除成员
    TeamRoleAssign,            // 分配角色
    TeamSettingsEdit,          // 修改团队设置
    
    // 同步操作
    SyncPush,                  // 推送变更到仓库
    SyncPull,                  // 拉取最新配置
```

### 1.3 服务器组权限（按组授权）

```json
// team.json 中的权限映射
{
  "role_overrides": {
    "user-uuid-123": {
      "groups": {
        "production": ["ServerConnect"],
        "staging": ["ServerConnect", "ServerEdit", "ServerViewCredentials"]
      }
    }
  }
}
```

允许管理员为特定成员指定在特定服务器组上的权限覆盖。例如：开发者默认只能连接，但在 staging 组有编辑权限。

### 1.4 权限检查

**新文件**: `src-tauri/src/team/permission.rs`

```rust
/// 检查当前用户是否有指定能力
pub fn check_permission(
    team_config: &TeamConfig,
    user_id: &str,
    capability: Capability,
    group_id: Option<&str>,
) -> bool {
    let member = team_config.members.iter()
        .find(|m| m.id == user_id)
        .unwrap_or_return(false);
    
    let role = &team_config.roles[&member.role];
    
    // 检查组级别覆盖
    if let Some(group) = group_id {
        if let Some(overrides) = team_config.role_overrides.get(user_id) {
            if let Some(group_caps) = overrides.groups.get(group) {
                return group_caps.contains(&capability);
            }
        }
    }
    
    // 回退到角色默认权限
    role.capabilities.contains(&capability)
}
```

**前端权限检查**:

```typescript
// src/composables/useTeamPermission.ts
export function useTeamPermission() {
  const teamStore = useTeamStore();
  
  function can(capability: Capability, groupId?: string): boolean {
    if (!teamStore.isTeamMode) return true; // 非团队模式不限制
    return teamStore.checkPermission(capability, groupId);
  }
  
  return { can };
}

// 在组件中使用
const { can } = useTeamPermission();
// v-if="can('ServerConnect', server.groupId)"
// :disabled="!can('ServerEdit')"
```

## 三、凭证隔离

### 3.1 问题

v0.30.0 中，共享服务器的凭证使用团队密钥加密。但拥有团队密钥的成员可以解密所有凭证。我们需要更细粒度的控制。

### 3.2 方案：凭证代理模式

不直接向客户端发送密码明文，而是使用"连接代理"模式：

```
方案 A（本迭代实现 - 客户端权限控制）:
- 凭证仍使用团队密钥加密存储在 Git 仓库中
- 客户端根据角色权限决定是否解密并显示
- ServerViewCredentials 权限控制明文可见性
- 无此权限的成员仍可连接（自动解密传递给 SSH 层，但不显示在 UI）

方案 B（未来 - 密钥分层加密）:
- 凭证使用角色密钥加密，非 admin 无法解密
- 需要密钥管理基础设施，复杂度高
- 留到 v1.0+ 企业版
```

### 3.3 实现

```rust
/// 获取共享服务器凭证（受权限控制）
#[tauri::command]
pub fn team_get_credentials(
    state: tauri::State<'_, AppState>,
    server_id: String,
    purpose: String,  // "connect" | "view"
) -> Result<Option<Credentials>, String> {
    let team_config = load_team_config(&state)?;
    let user_id = get_current_user_id(&state)?;
    let server = get_shared_server(&state, &server_id)?;
    
    match purpose.as_str() {
        "connect" => {
            // 连接时总是允许（只要有 ServerConnect 权限）
            check_permission(&team_config, &user_id, Capability::ServerConnect, server.group_id.as_deref())
                .then(|| decrypt_credentials(&server))
                .transpose()
        }
        "view" => {
            // 查看明文需要额外权限
            check_permission(&team_config, &user_id, Capability::ServerViewCredentials, server.group_id.as_deref())
                .then(|| decrypt_credentials(&server))
                .transpose()
        }
        _ => Err("Invalid purpose".into())
    }
}
```

## 四、成员管理

### 4.1 邀请流程

```
Admin 操作:
1. 点击 "邀请成员"
2. 生成邀请链接（包含 Git 仓库 URL + 加密的团队密钥片段）
3. 邀请链接格式: termex://team/join?repo=xxx&token=yyy
4. 发送给被邀请者（通过任何渠道）

被邀请者操作:
1. 点击邀请链接 / 手动输入
2. 输入团队密码（验证身份）
3. Termex 自动 clone 仓库 + 解密配置
4. 加入团队
```

### 4.2 移除成员

```rust
#[tauri::command]
pub fn team_remove_member(
    state: tauri::State<'_, AppState>,
    member_id: String,
) -> Result<(), String> {
    // 1. 检查当前用户有 TeamRemove 权限
    // 2. 从 team.json 移除成员
    // 3. Git commit + push
    // 4. 记录审计日志
    // 注意：被移除成员的本地 clone 仍然存在
    // 但他们下次 pull 会发现自己不在成员列表中
    // 客户端应拒绝显示共享资源
}
```

### 4.3 密码轮换

当成员被移除时，建议管理员轮换团队密码：

```
⚠️ 成员 "Bob" 已被移除。
建议轮换团队密码以确保安全性。
[立即轮换] [稍后提醒] [忽略]
```

轮换流程：
1. 管理员设置新密码
2. 所有凭证用新密钥重新加密
3. 其他在线成员下次 pull 时需要输入新密码
4. 旧密码失效

### 5.1 冲突场景

当两个成员同时修改同一服务器配置并 push 时，第二个 push 会失败。需要先 pull 并解决冲突。

### 5.3 冲突解决策略

```rust
pub enum ConflictStrategy {
    KeepLocal,       // 保留本地版本
    UseRemote,       // 使用远程版本
    Manual(Value),   // 手动合并后的值
    Skip,            // 跳过，保持冲突状态
}

pub struct ConflictItem {
    pub entity_type: String,    // "server", "snippet", "proxy"
    pub entity_id: String,
    pub entity_name: String,
    pub local_value: serde_json::Value,
    pub remote_value: serde_json::Value,
    pub conflicting_fields: Vec<String>,
    pub local_modified_by: String,
    pub remote_modified_by: String,
    pub local_modified_at: String,
    pub remote_modified_at: String,
}
```

1. `cd src-tauri && cargo test --release` — 所有测试通过
2. `cd src-tauri && cargo clippy` — 无警告
3. 创建团队，设置 4 个角色成员 → 各角色权限正确生效
4. developer 角色无法查看密码明文，但可以连接服务器
5. 两个成员同时修改同一服务器 → 冲突 UI 正确显示
6. 导出审计报告（HTML/CSV） → 格式正确，包含所有事件
7. 移除成员 → 提示密码轮换 → 轮换后被移除成员无法访问
8. 组级别权限覆盖 → developer 在 staging 组有额外权限

## 十一、风险与缓解

| 风险 | 严重性 | 缓解措施 |
|------|--------|---------|
| 客户端权限可被绕过（修改代码） | 中 | 凭证加密是最终防线；说明这不是零信任模型 |
| 团队密码泄露 | 高 | 支持密码轮换；审计凭证访问 |
| Git 合并冲突频繁 | 低 | 冲突 UI + 自动合并非冲突字段 |
| 大团队性能 | 低 | Git 仓库天然支持大量文件；客户端缓存 |
| 离线成员数据过期 | 低 | 上线自动同步 + 冲突提示 |
