# v0.40.0 — Mobile Foundation: 移动端基础 + 最小 SSH 客户端

> 状态：**计划中**
> 依赖：v0.35.0 (桌面端 v1.0 稳定版)
> 关联：移动端路线图第 1 迭代（共 4 迭代：v0.40.0 → v0.42.0 → v0.44.0 → v0.46.0）

---

## 背景

### 战略定位

以 SSH 协议为底座，打造 AI 时代永不断线的云端智能工作平台。

**移动端是实现这一战略的核心** — 用户应该能从任何设备、任何地点连接到云端工作空间。当前 Termex v0.35.0 是 100% 桌面端应用，需要从零搭建移动端支持。

### 当前状态评估

| 层级 | 移动就绪度 | 说明 |
|------|-----------|------|
| SSH 协议 (russh) | 95% | 纯 Rust，ARM64 兼容，无需改动 |
| 加密层 (ring + argon2) | 95% | 纯 Rust，跨平台兼容 |
| 数据库 (rusqlite + SQLCipher) | 95% | ARM64 兼容，bundled 编译 |
| 云端 AI 调用 | 95% | reqwest HTTP，无平台依赖 |
| SFTP 操作 | 95% | 纯 Rust russh-sftp |
| Tauri IPC 框架 | 80% | Tauri v2 原生支持移动端 |
| 本地 PTY (portable-pty) | 0% | 不支持 iOS/Android |
| 本地 AI (llama-server) | 0% | iOS 沙箱禁止子进程 |
| OS 凭据 (keyring) | 0% | 缺少 iOS/Android 后端 |
| 系统 API (clipboard/file/url) | 0% | arboard/rfd/open 均为桌面专属 |
| UI 布局 | 0% | 鼠标拖拽、固定多栏、无响应式 |
| xterm.js WebGL | 30% | 移动 WebView 不稳定，需降级 Canvas |

### 本迭代目标

在 iPhone / Android 上实现最小可用 SSH 客户端（iPad 架构预留，本迭代不实现）：

1. 打开应用 → 看到服务器列表
2. 点击服务器 → SSH 连接建立 → 全屏终端
3. 软键盘弹出 → 输入命令 → 执行
4. 断开连接 → 返回服务器列表

**不包含**: 服务器 CRUD、SFTP、AI、设置（留给后续迭代）

## 一、Cargo Feature Gates

### 1.1 问题

当前 `Cargo.toml` 中的 `portable-pty`、`arboard`、`rfd`、`open`、`keyring` 无法为 iOS/Android 编译。需要将这些依赖设为可选，并通过 feature flag 控制。

### 1.2 方案

引入 `desktop` feature（默认开启），桌面端行为不变，移动端编译时不包含这些依赖。

**修改文件**: `src-tauri/Cargo.toml`

```toml
[features]
default = ["desktop"]
desktop = [
    "dep:portable-pty",
    "dep:arboard",
    "dep:rfd",
    "dep:open",
    "dep:keyring"
]

[dependencies]
portable-pty = { version = "0.8", optional = true }
arboard = { version = "3", optional = true }
rfd = { version = "0.17.2", optional = true }
open = { version = "5", optional = true }
keyring = { version = "3", features = ["apple-native", "windows-native", "sync-secret-service"], optional = true }
```

### 1.3 原则

- Feature gate 尽量在**模块边界**（整文件/整模块），不在函数内部散落
- 桌面端默认 `desktop` feature 开启，行为完全不变
- CI 必须同时验证 `cargo build`（desktop）和 `cargo build --no-default-features`（mobile）

## 二、条件编译改造

### 2.1 整模块 Gate

以下模块整体用 `#[cfg(feature = "desktop")]` 包裹：

| 模块 | 原因 |
|------|------|
| `src-tauri/src/local_pty.rs` | 依赖 `portable-pty`，移动端无本地终端 |
| `src-tauri/src/local_ai/mod.rs` + 4 个子模块 | 依赖子进程管理，iOS 沙箱禁止 |
| `src-tauri/src/commands/menu.rs` | 原生菜单，移动端无 menubar |

### 2.2 commands/mod.rs 条件声明

```rust
#[cfg(feature = "desktop")]
pub mod local_ai;
#[cfg(feature = "desktop")]
pub mod menu;
// local_pty 命令也需要 gate
```

### 2.3 lib.rs 改造

`run()` 函数中需要条件编译的部分：

| 代码段 | Gate 方式 |
|--------|----------|
| `local_pty::PtyRegistry::new()` + `.manage()` | `#[cfg(feature = "desktop")]` |
| `build_menu()` + `app.set_menu()` + 菜单事件处理 | `#[cfg(feature = "desktop")]` |
| llama-server cleanup (`pkill` / `taskkill`) | `#[cfg(feature = "desktop")]` |
| invoke_handler 中桌面专属命令注册 | 条件编译或运行时空实现 |
| `on_window_event` 中 PTY 清理逻辑 | `#[cfg(feature = "desktop")]` |

### 2.4 state.rs 改造

`AppState` 中以下字段需要条件编译：

```rust
#[cfg(feature = "desktop")]
pub llama_server: tokio::sync::RwLock<LlamaServerState>,

#[cfg(feature = "desktop")]
pub active_downloads: tokio::sync::RwLock<HashMap<String, oneshot::Sender<()>>>,
```

### 2.5 keychain/mod.rs 改造

`keyring` crate 使用处 gate 为 `#[cfg(feature = "desktop")]`，移动端 `is_available()` 返回 `false`，走已有的 AES-256-GCM + SQLCipher 回退路径。

### 2.6 commands/clipboard.rs 改造

`arboard` 使用处 gate 为 `#[cfg(feature = "desktop")]`，移动端返回空字符串或错误提示。

### 2.7 commands/local_fs.rs 改造

`rfd` 和 `open` 使用处 gate 为 `#[cfg(feature = "desktop")]`，移动端相关函数返回 `Err("Not supported on mobile")`。

### 2.8 paths.rs 扩展

新增 iOS/Android 路径解析：

```rust
#[cfg(target_os = "ios")]
{
    // iOS: ~/Library/Application Support/termex/
    // 通过 dirs crate 或 NSSearchPathForDirectoriesInDomains FFI
}

#[cfg(target_os = "android")]
{
    // Android: app-internal storage
    // 通过 dirs crate 或 Tauri 提供的 app data path
}
```

## 四、前端平台检测

### 4.1 usePlatform.ts

**新建文件**: `src/composables/usePlatform.ts`

提供响应式平台检测信号：

```typescript
interface PlatformInfo {
  isMobile: boolean;     // iPhone / Android phone
  isTablet: boolean;     // iPad / Android tablet
  isDesktop: boolean;    // macOS / Windows / Linux
  isIOS: boolean;
  isAndroid: boolean;
  platform: 'ios' | 'android' | 'macos' | 'windows' | 'linux';
}
```

检测方式：
1. 优先使用 Tauri API (`@tauri-apps/api/os`)
2. 回退到 User Agent 检测
3. 视口宽度辅助判断 tablet vs phone（768px 分界）

### 4.2 main.ts 平台 class

在 `document.documentElement` 上添加平台 class，供 CSS 条件样式使用：

```typescript
// 现有: 'platform-mac' | 'platform-win' | 'platform-linux'
// 新增: 'platform-ios' | 'platform-android' | 'platform-mobile' | 'platform-tablet'
```

## 六、终端移动端适配

### 6.1 渲染器

**修改文件**: `src/composables/useTerminal.ts`

移动端跳过 WebGL addon，仅用 Canvas 渲染器：

```typescript
// 现有逻辑（useTerminal.ts:65-71）
if (!isMobile) {
  try {
    webglAddon = new WebglAddon();
    terminal.loadAddon(webglAddon);
  } catch {
    webglAddon = null;
  }
}
// 移动端自动使用 Canvas 回退
```

原因：WebGL 在 WKWebView (iOS) 和 Android WebView 中有已知的 context-loss 问题，尤其在应用切后台时。Canvas 性能对移动端 32ms 目标延迟足够。

### 6.2 键盘处理

`attachCustomKeyEventHandler` 移动端调整：

- 跳过桌面快捷键（Cmd+C/V、Ctrl+Shift+C 等）
- 返回 `true`（pass-through）让原生 WebView 处理
- IME 保护逻辑保留（`ev.isComposing || ev.keyCode === 229`）
- 自动补全相关快捷键在移动端通过工具栏按钮触发，非键盘事件

### 6.3 视口调整

当软键盘弹出时，终端需要缩小以适应可见区域：

```typescript
if (window.visualViewport) {
  window.visualViewport.addEventListener('resize', () => {
    // 调整终端容器高度 = visualViewport.height
    // 触发 fitAddon.fit()
  });
}
```

### 6.4 触摸交互

基础触摸交互（本迭代仅实现基本功能）：
- 点击终端 → 聚焦软键盘
- 长按 → 选择文本（由 WebView 原生处理）
- 上下滑动 → 滚动终端输出（xterm.js 内置支持）

高级手势（双指缩放、滑动切换等）留给迭代 4。

### 6.5 Android 返回键处理

Android 硬件/手势返回键是核心交互，必须从首个迭代就正确处理：

| 当前页面 | 返回键行为 |
|---------|-----------|
| 终端页（已连接） | 弹出确认对话框："断开连接并返回？" |
| 终端页（未连接） | 返回服务器列表 |
| 服务器列表 | 退出应用（或最小化到后台） |
| 设置页 | 返回上一页 |

实现方式：监听 Tauri 的 `tauri://back` 事件或 WebView 的 `popstate` / `backbutton` 事件。

## 九、风险与缓解

### 9.1 xterm.js 移动端 WebView 兼容性（高风险）

**风险**: xterm.js 主要面向桌面浏览器，移动 WebView 中触摸输入、软键盘聚焦、Canvas 性能可能有未知问题。

**缓解策略**:
1. **第一周先做 PoC** — 在 iOS 模拟器中运行最小 xterm.js 终端，验证：
   - `terminal.focus()` 能否唤起软键盘
   - Canvas 渲染帧率是否达到 30fps
   - 触摸事件是否正常传递
2. **备选方案** — 若 xterm.js 不可行，使用原生终端视图 (Swift WKWebView + 自定义 JS bridge 或 SwiftUI Terminal)

### 9.2 软键盘集成（高风险）

**风险**: WebView 虚拟键盘可能无法正确调整视口，xterm.js 内部 textarea 可能无法获得焦点。

**缓解策略**:
1. 使用 `visualViewport` API 检测键盘高度
2. 若默认 textarea 方案失败，注入可见 `<input>` 代理按键到 xterm.js
3. 务必在真机测试，不仅依赖模拟器

### 9.3 Feature Gate 破坏桌面构建（中风险）

**风险**: 引入 `#[cfg(feature = "desktop")]` 可能误伤桌面构建。

**缓解策略**:
1. Gate 尽量在模块边界（整文件/整模块）
2. CI 同时运行 desktop + mobile 两种构建
3. 所有现有测试必须在 desktop feature 下全部通过

### 9.4 Tauri v2 移动端成熟度（中风险）

**风险**: Tauri v2 移动端生态较新，可能有边缘问题。

**缓解策略**:
1. 锁定 Tauri v2 具体版本（当前 `version = "2"` 范围太宽）
2. 关注 Tauri GitHub mobile 标签的 issues
3. IPC invoke + events 是 Tauri 移动端最成熟的路径
