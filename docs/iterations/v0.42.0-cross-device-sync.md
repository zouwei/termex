# v0.42.0 — Cross-Device Sync: 跨设备配置同步

> 状态：**计划中**
> 依赖：v0.40.0 (移动端基础)
> 关联：移动端路线图第 2 迭代

---

## 背景

### 战略需求

Termex 的战略定位是"以 SSH 协议为底座，打造 AI 时代永不断线的云端智能工作平台"。移动端（v0.40.0）实现后，用户面临核心痛点：**桌面端已配置的数十台服务器如何同步到移动端？**

当前仅支持 `.termex` 加密文件导出/导入，存在以下局限：

| 现有能力 | 局限 |
|---------|------|
| 加密文件导出 (.termex) | 手动操作，需要文件传输（AirDrop/邮件等） |
| 密码/密钥排除 | 导出文件**不含**任何加密凭据，导入后需逐个重新输入 |
| 单向操作 | 无法双向同步，无增量更新 |
| 数据不完整 | 不含代理配置、AI 供应商、连接链路、SSH 密钥 |

### 目标

实现局域网内 P2P 跨设备配置同步：

1. **零配置发现** — 同一局域网内的 Termex 设备自动发现彼此
2. **完整数据同步** — 服务器、分组、代理、端口转发、连接链路、AI 供应商、设置、SSH 密钥
3. **凭据安全传输** — 密码/密钥通过配对密码端到端加密传输
4. **双向同步** — PC → 移动端 / 移动端 → PC，支持增量同步
5. **无云服务依赖** — 纯局域网 P2P，符合 Termex "纯本地应用" 定位

## 一、技术方案：P2P 局域网同步

### 1.1 方案评估

| 方案 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| **P2P 局域网** | 无云服务、隐私最强、速度快、离线可用 | 需同网络、AP 隔离可能阻断 | **采用** |
| 云端同步 | 随时随地、无网络限制 | 违背"纯本地"定位、需后端成本、隐私风险 | 不采用 |
| iCloud/Google Drive | 系统原生自动同步 | 平台锁定、跨平台不通、实现复杂 | 不采用 |
| 文件传输 | 最简单 | 手动、无凭据、无增量 | 已有（保留） |

### 1.2 核心架构

**桌面端必须是唯一的服务器**。原因：iOS 应用无法可靠运行后台 HTTP 服务器（沙箱限制 + 30 秒后台挂起）。

```
桌面端 (Server)                        移动端 (Client)
┌────────────────────────┐            ┌────────────────────────┐
│  axum HTTP Server      │◄── LAN ───│  reqwest HTTP Client   │
│  (port 17000-17099)    │            │                        │
├────────────────────────┤            ├────────────────────────┤
│  mDNS Publisher        │──广播────▶ │  mDNS Scanner          │
│  _termex-sync._tcp     │            │  (自动发现桌面端)       │
├────────────────────────┤            ├────────────────────────┤
│  Sync Engine           │            │  Sync Engine           │
│  ├ 配对管理             │            │  ├ 配对管理             │
│  ├ 冲突解决             │            │  ├ 冲突解决             │
│  └ 增量计算             │            │  └ 增量计算             │
├────────────────────────┤            ├────────────────────────┤
│  SQLCipher DB (主)     │            │  SQLCipher DB (副)     │
└────────────────────────┘            └────────────────────────┘
```

**双向同步流程**：无论是 PC→Mobile 还是 Mobile→PC，都是移动端主动连接桌面端 HTTP API，推送或拉取数据。

### 1.3 技术栈选型

| 组件 | 选型 | 原因 |
|------|------|------|
| HTTP 服务器 | `axum 0.7` | 基于 tokio（已有），轻量，类型安全 |
| 设备发现 | `mdns-sd 0.11` | 纯 Rust mDNS/DNS-SD，跨平台，支持 iOS/Android |
| 传输加密 | 复用 `crypto/aes.rs` | AES-256-GCM，已有且经过测试 |
| 密钥派生 | 复用 `crypto/kdf.rs` | Argon2id，已有 |
| 序列化 | `serde_json` | 已有 |
| 端口范围 | 17000-17099 | 避开 llama-server (15000-16000) |

## 二、设备发现（mDNS/DNS-SD）

### 2.1 服务注册

桌面端启动同步服务时，通过 mDNS 广播服务：

```
Service Type: _termex-sync._tcp.local
Instance Name: "Termex - <设备名>"
Port: 17000 (动态分配)
TXT Records:
  version=1
  device_id=<uuid>
  device_name=<hostname>
  app_version=0.42.0
```

### 2.2 服务发现

移动端扫描局域网中的 `_termex-sync._tcp` 服务：

- 打开同步页面时自动开始扫描
- 显示发现的桌面端设备列表（设备名 + IP + 版本号）
- 支持手动输入 IP:Port（应对 mDNS 不可用的网络环境）
- 扫描超时 10 秒，期间显示加载动画

### 2.3 mDNS 不可用的回退方案

某些网络环境（企业 WiFi、AP 隔离、跨 VLAN）mDNS 不可达：

| 回退方案 | 说明 |
|---------|------|
| 手动输入 IP | 用户在移动端手动输入桌面端 IP:Port |
| QR 码扫描 | 桌面端生成 QR 码（含 IP + Port + 配对信息），移动端扫描 |

## 三、设备配对与安全

### 3.1 首次配对流程

```
桌面端                              移动端
  │                                   │
  │  1. 用户点击"启用同步"              │
  │  2. 生成 6 位配对码                │
  │  3. 屏幕显示配对码                 │
  │                                   │
  │              ◄──── 4. 扫描发现桌面端
  │              ◄──── 5. 用户输入配对码
  │                                   │
  │  6. 验证配对码                     │
  │  7. 交换设备 ID + 公钥             │
  │  8. 派生共享密钥                   │
  │  9. 持久化配对关系                  │
  │ ────── 10. 配对成功 ──────▶        │
  │                                   │
  │  后续同步自动认证（无需再输入）       │
```

### 3.2 安全机制

| 层级 | 机制 |
|------|------|
| 传输层 | AES-256-GCM 端到端加密（共享密钥派生自配对码 + 设备密钥） |
| 认证层 | 设备 ID + HMAC-SHA256 签名（配对后自动） |
| 配对码 | 6 位数字，3 分钟有效期，单次使用 |
| 密钥派生 | 配对码 → Argon2id → 共享密钥 |

### 3.3 凭据传输安全

**核心问题**：当前 `.termex` 导出故意排除所有密码/密钥。P2P 同步需要支持凭据传输。

**方案**：配对设备间端到端加密传输凭据

1. 源端从 Keychain/SQLCipher 解密凭据到内存
2. 用共享密钥 AES-256-GCM 加密凭据
3. 通过 HTTPS/HTTP 传输加密数据
4. 目标端解密后存入本地 Keychain/SQLCipher
5. 内存中的明文凭据立即 zeroize

**用户可选**：同步设置中可关闭凭据同步（仅同步结构信息，凭据手动输入）。

### 3.4 配对数据持久化

数据库新增 `sync_devices` 表：

```sql
CREATE TABLE sync_devices (
    id TEXT PRIMARY KEY,                -- 设备 UUID
    name TEXT NOT NULL,                 -- 设备名称
    platform TEXT NOT NULL,             -- ios / android / macos / windows / linux
    shared_key_enc BLOB,               -- 加密存储的共享密钥
    last_sync_at TEXT,                  -- 上次同步时间 (ISO 8601)
    last_sync_version INTEGER DEFAULT 0, -- 上次同步的数据版本号
    paired_at TEXT NOT NULL,            -- 配对时间
    app_version TEXT                    -- 对端应用版本
);
```

## 四、同步协议

### 4.1 HTTP API

桌面端 axum 服务器提供以下 REST API：

| Method | Path | 说明 |
|--------|------|------|
| GET | `/api/v1/sync/status` | 服务状态（版本号、设备名） |
| POST | `/api/v1/sync/pair` | 设备配对（提交配对码） |
| GET | `/api/v1/sync/pull` | 拉取配置（增量，带 since_version 参数） |
| POST | `/api/v1/sync/push` | 推送配置（增量，移动端 → 桌面端） |
| DELETE | `/api/v1/sync/unpair` | 解除配对 |

所有请求头包含：
```
X-Device-Id: <uuid>
X-Signature: <HMAC-SHA256(body, shared_key)>
X-Timestamp: <unix_ms>
```

### 4.2 同步数据范围

| 数据类型 | 包含 | 说明 |
|---------|------|------|
| servers | 是 | 全部字段 |
| groups | 是 | 分组结构 |
| port_forwards | 是 | 端口转发规则 |
| proxies | 是 | 代理配置 |
| connection_chain | 是 | 连接链路 |
| ai_providers | 是 | AI 供应商配置 |
| ssh_keys | 是 | SSH 密钥（公钥 + 文件路径引用） |
| settings | 是 | 应用设置（key-value） |
| known_hosts | 是 | 已知主机指纹 |
| **凭据** (密码/密钥/API Key) | **可选** | 用户可选是否同步凭据 |
| audit_log | 否 | 审计日志不同步 |
| local_ai models | 否 | 移动端不支持本地 AI |

### 4.3 增量同步机制

基于版本号的增量同步：

1. 每次数据变更（增/删/改），全局版本号 +1
2. 每条记录携带 `sync_version` 字段（记录最后修改的版本号）
3. 拉取时传入 `since_version`，服务端返回版本号大于该值的所有记录
4. 推送时客户端发送变更记录，服务端合并

数据库新增 `sync_metadata` 表：

```sql
CREATE TABLE sync_metadata (
    key TEXT PRIMARY KEY,
    value TEXT NOT NULL
);
-- key: 'global_version' → 当前全局版本号
-- key: 'device_id' → 本设备 UUID
```

各业务表新增字段：

```sql
ALTER TABLE servers ADD COLUMN sync_version INTEGER DEFAULT 0;
ALTER TABLE servers ADD COLUMN sync_deleted INTEGER DEFAULT 0;  -- 软删除标记
-- groups, port_forwards, proxies, connection_chain, ai_providers, ssh_keys, settings 同理
```

### 4.4 冲突解决

同步冲突策略：**Last-Write-Wins (LWW)**

| 场景 | 处理 |
|------|------|
| 同一服务器两端都修改 | 比较 `updated_at` 时间戳，取最新 |
| 一端删除、另一端修改 | 删除优先（tombstone 保留 30 天） |
| 新增同 ID 记录 | UUID 冲突概率极低，但若发生则以时间戳为准 |
| 设置冲突 | Last-Write-Wins（按 key 粒度） |

用户可选策略（未来扩展）：
- 始终以桌面端为准
- 始终以移动端为准
- 逐条确认

### 5.2 axum 服务器

**新建文件**: `src-tauri/src/sync/server.rs`

```rust
use axum::{Router, routing::{get, post, delete}, extract::State};

pub async fn start_sync_server(db: Arc<Database>, port: u16) -> Result<(), String> {
    let app = Router::new()
        .route("/api/v1/sync/status", get(handle_status))
        .route("/api/v1/sync/pair", post(handle_pair))
        .route("/api/v1/sync/pull", get(handle_pull))
        .route("/api/v1/sync/push", post(handle_push))
        .route("/api/v1/sync/unpair", delete(handle_unpair))
        .layer(/* HMAC 签名验证中间件 */)
        .with_state(sync_state);

    let listener = tokio::net::TcpListener::bind(("0.0.0.0", port)).await?;
    axum::serve(listener, app).await?;
    Ok(())
}
```

## 八、同步流程详解

### 8.1 首次同步（PC → 移动端）

```
1. 桌面端: 设置 → 同步 → 启用同步服务
2. 桌面端: 生成配对码 "482917"，显示在屏幕上
3. 移动端: 设置 → 同步 → 扫描发现 "Termex - MacBook Pro"
4. 移动端: 点击设备 → 输入配对码 "482917"
5. 握手: 交换设备 ID，派生共享密钥
6. 移动端: 点击 "拉取配置"
7. 桌面端: 计算全量快照 → AES 加密 → HTTP 返回
8. 移动端: 解密 → 写入本地 DB → 显示结果 "新增 23 台服务器"
```

### 8.2 增量同步（日常使用）

```
1. 移动端: 打开同步页 → 自动发现已配对桌面端
2. 移动端: GET /api/v1/sync/pull?since_version=42
3. 桌面端: 返回版本 42 之后的变更记录
4. 移动端: 合并变更 → 更新本地 DB
5. 移动端: POST /api/v1/sync/push (推送移动端变更)
6. 桌面端: 合并移动端变更 → 返回冲突报告（如有）
```

### 8.3 反向同步（移动端 → PC）

移动端新建/修改的服务器推送到桌面端：

```
1. 移动端: POST /api/v1/sync/push { changes: [...] }
2. 桌面端: 验证签名 → 解密 → 合并到 DB
3. 桌面端: 返回 { merged: 3, conflicts: 0 }
```

## 九、风险与缓解

### 9.1 mDNS 不可达（中风险）

**风险**: 企业 WiFi 可能禁用 mDNS 广播，AP 隔离阻断设备间通信。

**缓解**: 提供手动输入 IP:Port 和 QR 码扫描两种回退方案。

### 9.2 凭据泄露（高风险）

**风险**: P2P 传输中凭据被截获。

**缓解**: 端到端 AES-256-GCM 加密，密钥通过 Argon2id 从配对码派生。即使网络被监听，无配对码无法解密。用户可选关闭凭据同步。

### 9.3 数据冲突（低风险）

**风险**: 两端同时修改同一服务器配置。

**缓解**: Last-Write-Wins 策略，基于 `updated_at` 时间戳。冲突发生概率低（配置不频繁变更）。

### 9.4 axum 服务器安全（中风险）

**风险**: 同步 HTTP 服务器暴露在局域网，可能被未授权设备访问。

**缓解**: 所有 API 请求必须携带有效 HMAC 签名（仅已配对设备持有共享密钥）。未配对设备只能访问 `/api/v1/sync/pair`（需配对码）。

## 验证标准

### 配对验证

- [ ] 桌面端启用同步服务，mDNS 广播正常
- [ ] 移动端扫描发现桌面端设备
- [ ] 移动端手动输入 IP:Port 可连接
- [ ] 配对码输入正确 → 配对成功
- [ ] 配对码输入错误 → 拒绝 + 错误提示
- [ ] 配对码 3 分钟过期后无效
- [ ] 已配对设备列表正确显示

### 同步验证

- [ ] PC → 移动端：全量拉取，服务器/分组/代理/链路完整
- [ ] PC → 移动端：增量拉取，仅传输变更数据
- [ ] 移动端 → PC：移动端新建服务器推送到桌面端
- [ ] 凭据同步开启时：密码正确传输并可用
- [ ] 凭据同步关闭时：密码字段为空
- [ ] 冲突场景：两端修改同一服务器，Last-Write-Wins 生效

### 安全验证

- [ ] 未配对设备无法访问 /pull 和 /push API
- [ ] 传输数据 AES-256-GCM 加密
- [ ] 内存中凭据明文自动 zeroize
- [ ] `cargo test --release --test test_sync` 全部通过

### 回归验证

- [ ] 桌面端所有功能无回归
- [ ] 现有 `.termex` 文件导出/导入不受影响
- [ ] 数据库迁移 V12 向后兼容
