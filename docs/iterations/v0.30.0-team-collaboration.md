## 背景

### 当前状态

Termex 是一个纯单用户桌面应用（符合 "pure local app" 定位）。团队场景下，每位工程师需要独立配置所有服务器连接 — 即使是同一团队管理的相同服务器集群。

**现实痛点**：

| 场景 | 当前解法 | 问题 |
|------|---------|------|
| 新人入职，需要配置 30+ 服务器 | 老成员导出 `.termex` 文件发送 | 手动操作，凭据不含在内需逐个填写 |
| 运维变更服务器 IP | 口头/群消息通知各人 | 不可靠，必有遗漏 |
| 团队积累的常用命令 snippets | 无共享机制 | 重复造轮子 |
| 服务器分类/分组结构 | 每人各自维护 | 命名/结构不统一 |
| 离职时凭据回收 | 依赖人工更换密码 | 无法确认是否遗漏 |

### 设计哲学

**Git 即后端** — 利用 Git 仓库作为团队配置的存储和同步介质。不引入云服务器、不引入 SaaS 组件。

**为什么是 Git？**

1. **无云依赖** — 完全符合 Termex "纯本地应用" 定位
2. **零成本** — 所有团队都有 GitHub/GitLab/Gitea
3. **版本历史** — 谁在什么时候改了什么，Git 天然记录
4. **离线优先** — 本地 clone，随时可用，联网时再 push/pull
5. **去中心化** — 无单点故障，任何成员都有完整副本
6. **安全** — 凭据用团队密钥加密后才入库，Git 服务器看不到明文
7. **生态成熟** — `git2` crate（libgit2 绑定）稳定可靠

### 目标

1. 团队创建/加入 — 初始化或 clone Git 仓库，设置团队密钥
2. 选择性共享 — per-server、per-snippet 粒度控制
3. 双向同步 — pull 最新 + push 本地变更，自动冲突解决
4. 角色控制 — admin/member/readonly 简单权限模型
5. 审计集成 — 团队操作写入 audit_log

## 二、团队密钥加密

### 2.1 密钥派生

团队密钥从团队密码短语（passphrase）通过 Argon2id 派生，与个人 master password 的 KDF 参数一致：

```rust
/// 从团队密码短语派生加密密钥
/// 使用固定 salt（存储在 team.json 中），确保所有成员从相同 passphrase 派生出相同密钥
pub fn derive_team_key(passphrase: &str, salt: &[u8; 16]) -> Zeroizing<[u8; 32]> {
    let params = argon2::Params::new(
        65536,  // m_cost: 64 MB
        3,      // t_cost: 3 iterations
        4,      // p_cost: 4 parallelism
        Some(32),
    ).expect("valid params");

    let argon2 = argon2::Argon2::new(
        argon2::Algorithm::Argon2id,
        argon2::Version::V0x13,
        params,
    );

    let mut key = Zeroizing::new([0u8; 32]);
    argon2.hash_password_into(
        passphrase.as_bytes(),
        salt,
        key.as_mut(),
    ).expect("hash");

    key
}
```

**Salt 管理**：
- 创建团队时随机生成 16 字节 salt，存入 `team.json` 的 `_salt` 字段（hex 编码）
- Salt 不是机密 — 即使仓库被泄露，无 passphrase 仍无法解密
- 所有团队成员使用相同 salt + passphrase → 派生出相同密钥

### 2.2 加密/解密

**新建文件**: `src-tauri/src/team/crypto.rs` (~100 行)

```rust
use ring::aead::{self, LessSafeKey, Nonce, UnboundKey, AES_256_GCM};
use zeroize::Zeroizing;

/// 用团队密钥加密字符串（AES-256-GCM）
/// 返回 base64 编码的 nonce + ciphertext + tag
pub fn team_encrypt(key: &[u8; 32], plaintext: &str) -> Result<String, TeamCryptoError> {
    let unbound = UnboundKey::new(&AES_256_GCM, key)
        .map_err(|_| TeamCryptoError::KeyError)?;
    let less_safe = LessSafeKey::new(unbound);

    // 随机 96-bit nonce
    let mut nonce_bytes = [0u8; 12];
    ring::rand::SystemRandom::new()
        .fill(&mut nonce_bytes)
        .map_err(|_| TeamCryptoError::RngError)?;
    let nonce = Nonce::assume_unique_for_key(nonce_bytes);

    let mut in_out = plaintext.as_bytes().to_vec();
    less_safe.seal_in_place_append_tag(nonce, aead::Aad::empty(), &mut in_out)
        .map_err(|_| TeamCryptoError::EncryptError)?;

    // nonce (12) + ciphertext + tag (16) → base64
    let mut output = nonce_bytes.to_vec();
    output.extend_from_slice(&in_out);
    Ok(base64::engine::general_purpose::STANDARD.encode(&output))
}

/// 用团队密钥解密 base64 编码的密文
pub fn team_decrypt(key: &[u8; 32], ciphertext_b64: &str) -> Result<String, TeamCryptoError> {
    let data = base64::engine::general_purpose::STANDARD
        .decode(ciphertext_b64)
        .map_err(|_| TeamCryptoError::DecodeError)?;

    if data.len() < 12 + 16 {
        return Err(TeamCryptoError::InvalidData);
    }

    let nonce_bytes: [u8; 12] = data[..12].try_into().unwrap();
    let nonce = Nonce::assume_unique_for_key(nonce_bytes);

    let unbound = UnboundKey::new(&AES_256_GCM, key)
        .map_err(|_| TeamCryptoError::KeyError)?;
    let less_safe = LessSafeKey::new(unbound);

    let mut in_out = data[12..].to_vec();
    let plaintext = less_safe.open_in_place(nonce, aead::Aad::empty(), &mut in_out)
        .map_err(|_| TeamCryptoError::DecryptError)?;

    String::from_utf8(plaintext.to_vec())
        .map_err(|_| TeamCryptoError::Utf8Error)
}

#[derive(Debug, thiserror::Error)]
pub enum TeamCryptoError {
    #[error("key error")]
    KeyError,
    #[error("RNG error")]
    RngError,
    #[error("encrypt error")]
    EncryptError,
    #[error("decrypt error")]
    DecryptError,
    #[error("base64 decode error")]
    DecodeError,
    #[error("invalid data")]
    InvalidData,
    #[error("UTF-8 error")]
    Utf8Error,
}
```

### 2.3 安全策略

| 规则 | 说明 |
|------|------|
| 团队密钥只在内存 | 启动后输入 passphrase → 派生密钥 → 内存持有 → 退出时 zeroize |
| 仓库中无明文凭据 | 所有 `*_enc` 字段都是 AES-256-GCM 密文 |
| Git 服务器不可信 | 即使 GitHub 被入侵，无 passphrase 无法解密 |
| 团队密钥与个人密钥独立 | 团队 passphrase ≠ 个人 master password |
| Passphrase 不可恢复 | 丢失 passphrase → 无法解密团队凭据 → 需要 admin 重置 |

### 3.2 Git 模块

**新建文件**: `src-tauri/src/team/git.rs` (~200 行)

```rust
use git2::{
    Cred, FetchOptions, PushOptions, RemoteCallbacks, Repository,
    ResetType, Signature,
};
use std::path::PathBuf;

/// 团队仓库操作
pub struct TeamRepo {
    path: PathBuf,
}

impl TeamRepo {
    /// 克隆远程仓库
    pub fn clone(url: &str, path: &PathBuf, credentials: &GitCredentials) -> Result<Self, TeamGitError> {
        let mut callbacks = RemoteCallbacks::new();
        setup_credentials(&mut callbacks, credentials);

        let mut fetch_opts = FetchOptions::new();
        fetch_opts.remote_callbacks(callbacks);

        let mut builder = git2::build::RepoBuilder::new();
        builder.fetch_options(fetch_opts);

        builder.clone(url, path)
            .map_err(|e| TeamGitError::CloneError(e.to_string()))?;

        Ok(Self { path: path.clone() })
    }

    /// 打开已有本地仓库
    pub fn open(path: &PathBuf) -> Result<Self, TeamGitError> {
        Repository::open(path)
            .map_err(|e| TeamGitError::OpenError(e.to_string()))?;
        Ok(Self { path: path.clone() })
    }

    /// Pull: fetch + merge (fast-forward only)
    pub fn pull(&self, credentials: &GitCredentials) -> Result<PullResult, TeamGitError> {
        let repo = Repository::open(&self.path)
            .map_err(|e| TeamGitError::OpenError(e.to_string()))?;

        // fetch origin/main
        let mut callbacks = RemoteCallbacks::new();
        setup_credentials(&mut callbacks, credentials);

        let mut fetch_opts = FetchOptions::new();
        fetch_opts.remote_callbacks(callbacks);

        let mut remote = repo.find_remote("origin")
            .map_err(|e| TeamGitError::FetchError(e.to_string()))?;
        remote.fetch(&["main"], Some(&mut fetch_opts), None)
            .map_err(|e| TeamGitError::FetchError(e.to_string()))?;

        // Fast-forward merge
        let fetch_head = repo.find_reference("FETCH_HEAD")
            .map_err(|e| TeamGitError::MergeError(e.to_string()))?;
        let fetch_commit = repo.reference_to_annotated_commit(&fetch_head)
            .map_err(|e| TeamGitError::MergeError(e.to_string()))?;

        let analysis = repo.merge_analysis(&[&fetch_commit])
            .map_err(|e| TeamGitError::MergeError(e.to_string()))?;

        if analysis.0.is_up_to_date() {
            return Ok(PullResult::UpToDate);
        }

        if analysis.0.is_fast_forward() {
            // 执行 fast-forward
            let refname = "refs/heads/main";
            let mut reference = repo.find_reference(refname)
                .map_err(|e| TeamGitError::MergeError(e.to_string()))?;
            reference.set_target(fetch_commit.id(), "fast-forward")
                .map_err(|e| TeamGitError::MergeError(e.to_string()))?;
            repo.set_head(refname)
                .map_err(|e| TeamGitError::MergeError(e.to_string()))?;
            repo.checkout_head(Some(git2::build::CheckoutBuilder::default().force()))
                .map_err(|e| TeamGitError::MergeError(e.to_string()))?;
            return Ok(PullResult::FastForward);
        }

        // 非 fast-forward → 需要冲突解决
        Ok(PullResult::Conflict)
    }

    /// Commit + Push
    pub fn commit_and_push(
        &self,
        message: &str,
        username: &str,
        credentials: &GitCredentials,
    ) -> Result<(), TeamGitError> {
        let repo = Repository::open(&self.path)
            .map_err(|e| TeamGitError::OpenError(e.to_string()))?;

        // Stage all changes
        let mut index = repo.index()
            .map_err(|e| TeamGitError::CommitError(e.to_string()))?;
        index.add_all(["*"].iter(), git2::IndexAddOption::DEFAULT, None)
            .map_err(|e| TeamGitError::CommitError(e.to_string()))?;
        index.write()
            .map_err(|e| TeamGitError::CommitError(e.to_string()))?;

        let oid = index.write_tree()
            .map_err(|e| TeamGitError::CommitError(e.to_string()))?;
        let tree = repo.find_tree(oid)
            .map_err(|e| TeamGitError::CommitError(e.to_string()))?;

        let sig = Signature::now(username, &format!("{}@termex.local", username))
            .map_err(|e| TeamGitError::CommitError(e.to_string()))?;

        let parent = repo.head().ok()
            .and_then(|h| h.peel_to_commit().ok());
        let parents: Vec<&git2::Commit> = parent.iter().collect();

        repo.commit(Some("HEAD"), &sig, &sig, message, &tree, &parents)
            .map_err(|e| TeamGitError::CommitError(e.to_string()))?;

        // Push
        let mut callbacks = RemoteCallbacks::new();
        setup_credentials(&mut callbacks, credentials);

        let mut push_opts = PushOptions::new();
        push_opts.remote_callbacks(callbacks);

        let mut remote = repo.find_remote("origin")
            .map_err(|e| TeamGitError::PushError(e.to_string()))?;
        remote.push(&["refs/heads/main"], Some(&mut push_opts))
            .map_err(|e| TeamGitError::PushError(e.to_string()))?;

        Ok(())
    }

    /// 获取仓库状态（是否有未提交变更）
    pub fn status(&self) -> Result<RepoStatus, TeamGitError> {
        let repo = Repository::open(&self.path)
            .map_err(|e| TeamGitError::OpenError(e.to_string()))?;
        let statuses = repo.statuses(None)
            .map_err(|e| TeamGitError::OpenError(e.to_string()))?;

        Ok(RepoStatus {
            has_changes: !statuses.is_empty(),
            changed_files: statuses.len(),
        })
    }
}

/// Git 认证凭据
pub enum GitCredentials {
    /// SSH key 认证
    SshKey { private_key_path: PathBuf, passphrase: Option<String> },
    /// HTTPS token 认证
    HttpsToken { token: String },
    /// HTTPS user/password
    HttpsUserPass { username: String, password: String },
}

fn setup_credentials(callbacks: &mut RemoteCallbacks, creds: &GitCredentials) {
    let creds = creds.clone();
    callbacks.credentials(move |_url, username_from_url, allowed_types| {
        match &creds {
            GitCredentials::SshKey { private_key_path, passphrase } => {
                Cred::ssh_key(
                    username_from_url.unwrap_or("git"),
                    None,
                    private_key_path,
                    passphrase.as_deref(),
                )
            }
            GitCredentials::HttpsToken { token } => {
                Cred::userpass_plaintext(token, "")
            }
            GitCredentials::HttpsUserPass { username, password } => {
                Cred::userpass_plaintext(username, password)
            }
        }
    });
}

#[derive(Debug)]
pub enum PullResult {
    UpToDate,
    FastForward,
    Conflict,
}

pub struct RepoStatus {
    pub has_changes: bool,
    pub changed_files: usize,
}

#[derive(Debug, thiserror::Error)]
pub enum TeamGitError {
    #[error("clone failed: {0}")]
    CloneError(String),
    #[error("open failed: {0}")]
    OpenError(String),
    #[error("fetch failed: {0}")]
    FetchError(String),
    #[error("merge failed: {0}")]
    MergeError(String),
    #[error("commit failed: {0}")]
    CommitError(String),
    #[error("push failed: {0}")]
    PushError(String),
}
```

## 四、同步引擎

### 4.1 同步流程

**新建文件**: `src-tauri/src/team/sync.rs` (~250 行)

同步 = pull remote → merge into local DB → push local changes

```
┌─────────────────────────────────────────────────────┐
│                    team_sync()                       │
├─────────────────────────────────────────────────────┤
│  1. git pull (fetch + fast-forward)                  │
│     ├─ UpToDate → skip to step 3                    │
│     ├─ FastForward → proceed to step 2              │
│     └─ Conflict → abort, notify user                │
│                                                      │
│  2. Import remote changes → local DB                 │
│     ├─ Read each servers/{id}.json                   │
│     ├─ Decrypt password_enc with team key            │
│     ├─ Compare with local: newer wins (updated_at)   │
│     ├─ Insert/update in servers table                │
│     └─ Mark as team_id = current team                │
│                                                      │
│  3. Export local shared changes → repo files         │
│     ├─ Query servers WHERE shared = 1                │
│     ├─ Encrypt password with team key                │
│     ├─ Write to servers/{id}.json                    │
│     └─ Same for groups, snippets, proxies            │
│                                                      │
│  4. git commit + push (if changes exist)             │
│     └─ Message: "sync: {username} @ {timestamp}"     │
└─────────────────────────────────────────────────────┘
```

### 4.2 冲突解决策略

采用 **Last Writer Wins (LWW)** 策略：`updated_at` 时间戳更新的版本覆盖旧版本。

```rust
/// 合并策略：比较 remote 和 local 的 updated_at，取更新者
fn merge_server(
    local: Option<&ServerRow>,
    remote: &SharedServerConfig,
    team_key: &[u8; 32],
) -> MergeDecision {
    match local {
        None => {
            // 本地不存在 → 导入 remote
            MergeDecision::ImportRemote
        }
        Some(local_row) => {
            let local_ts = parse_timestamp(&local_row.updated_at);
            let remote_ts = parse_timestamp(&remote.updated_at);

            if remote_ts > local_ts {
                MergeDecision::ImportRemote   // remote 更新 → 覆盖本地
            } else if local_ts > remote_ts {
                MergeDecision::ExportLocal    // 本地更新 → push 到 remote
            } else {
                MergeDecision::NoAction       // 相同 → 不动
            }
        }
    }
}

enum MergeDecision {
    ImportRemote,
    ExportLocal,
    NoAction,
}
```

### 4.3 共享服务器的凭据处理

Import 流程（remote → local）：

```rust
fn import_server(
    conn: &Connection,
    remote: &SharedServerConfig,
    team_key: &[u8; 32],
    keychain: &KeychainStore,
) -> Result<(), TeamSyncError> {
    // 1. 解密团队密文
    let password = remote.password_enc.as_ref()
        .map(|enc| team_decrypt(team_key, enc))
        .transpose()?;

    // 2. 存入本地 keychain（个人密钥保护）
    if let Some(ref pwd) = password {
        let keychain_id = format!("termex:ssh:password:{}", remote.id);
        keychain.store(&keychain_id, pwd)?;
    }

    // 3. 写入 servers 表（不含明文密码，只存 keychain_id）
    storage::server::upsert_shared(conn, remote, password.is_some())?;

    Ok(())
}
```

Export 流程（local → remote）：

```rust
fn export_server(
    conn: &Connection,
    server: &ServerRow,
    team_key: &[u8; 32],
    keychain: &KeychainStore,
    repo_path: &PathBuf,
) -> Result<(), TeamSyncError> {
    // 1. 从 keychain 读取明文密码
    let password = server.password_keychain_id.as_ref()
        .and_then(|kid| keychain.get(kid).ok());

    // 2. 用团队密钥加密
    let password_enc = password.as_ref()
        .map(|pwd| team_encrypt(team_key, pwd))
        .transpose()?;

    // 3. 构造 SharedServerConfig JSON
    let config = SharedServerConfig {
        id: server.id.clone(),
        name: server.name.clone(),
        host: server.host.clone(),
        port: server.port,
        username: server.username.clone(),
        auth_type: server.auth_type.clone(),
        password_enc,
        // ... other fields ...
        shared_by: current_username(),
        shared_at: crate::commands::recording::now_rfc3339(),
        updated_at: server.updated_at.clone(),
    };

    // 4. 写入 servers/{id}.json
    let file_path = repo_path.join("servers").join(format!("{}.json", server.id));
    std::fs::create_dir_all(file_path.parent().unwrap())?;
    let json = serde_json::to_string_pretty(&config)?;
    std::fs::write(&file_path, json)?;

    Ok(())
}
```

### 4.4 删除同步

当本地删除一个共享服务器时，需要在仓库中标记删除：

- 方式一：直接删除 `servers/{id}.json` 文件 → Git 追踪删除
- 方式二：在文件中添加 `"deleted": true` 标记（tombstone）

采用**方式一**（Git 原生支持文件删除，简单可靠）。其他成员 pull 后检测到文件消失 → 从本地 DB 删除对应记录。

### 8.1 teamStore

```typescript
import { defineStore } from 'pinia';
import { ref, computed } from 'vue';
import type { TeamStatus, TeamMember, TeamSyncResult, GitAuthConfig } from '@/types/team';
import { invoke } from '@/utils/tauri';

export const useTeamStore = defineStore('team', () => {
  const status = ref<TeamStatus>({
    joined: false,
    name: null,
    role: null,
    memberCount: 0,
    lastSync: null,
    hasPendingChanges: false,
    repoUrl: null,
  });

  const members = ref<TeamMember[]>([]);
  const syncing = ref(false);

  const isJoined = computed(() => status.value.joined);
  const isAdmin = computed(() => status.value.role === 'admin');
  const teamName = computed(() => status.value.name || '');
  const canPush = computed(() =>
    status.value.role === 'admin' || status.value.role === 'member'
  );

  /** 加载团队状态 */
  async function loadStatus() {
    status.value = await invoke<TeamStatus>('team_get_status');
  }

  /** 创建团队 */
  async function create(
    name: string,
    passphrase: string,
    repoUrl: string,
    gitAuth: GitAuthConfig,
  ) {
    await invoke('team_create', { name, passphrase, repoUrl, gitAuth });
    await loadStatus();
  }

  /** 加入团队 */
  async function join(
    repoUrl: string,
    passphrase: string,
    username: string,
    gitAuth: GitAuthConfig,
  ) {
    await invoke('team_join', { repoUrl, passphrase, username, gitAuth });
    await loadStatus();
  }

  /** 同步 */
  async function sync(): Promise<TeamSyncResult> {
    syncing.value = true;
    try {
      const result = await invoke<TeamSyncResult>('team_sync');
      await loadStatus();
      return result;
    } finally {
      syncing.value = false;
    }
  }

  /** 离开团队 */
  async function leave() {
    await invoke('team_leave');
    status.value = { joined: false, name: null, role: null, memberCount: 0, lastSync: null, hasPendingChanges: false, repoUrl: null };
    members.value = [];
  }

  /** 加载成员列表 */
  async function loadMembers() {
    members.value = await invoke<TeamMember[]>('team_list_members');
  }

  /** 修改成员角色 (admin only) */
  async function setMemberRole(username: string, role: string) {
    await invoke('team_set_role', { targetUsername: username, role });
    await loadMembers();
  }

  /** 移除成员 (admin only) */
  async function removeMember(username: string) {
    await invoke('team_remove_member', { targetUsername: username });
    await loadMembers();
  }

  return {
    status, members, syncing,
    isJoined, isAdmin, teamName, canPush,
    loadStatus, create, join, sync, leave,
    loadMembers, setMemberRole, removeMember,
  };
});
```

### 10.2 客户端强制执行

```rust
fn check_role(state: &AppState, required: &str) -> Result<(), String> {
    let role = state.db.with_conn(|conn| {
        storage::settings::get(conn, "team_role")
    }).map_err(|e| e.to_string())?
        .ok_or("not in a team")?;

    let role_level = match role.as_str() {
        "admin" => 3,
        "member" => 2,
        "readonly" => 1,
        _ => 0,
    };
    let required_level = match required {
        "admin" => 3,
        "member" => 2,
        "readonly" => 1,
        _ => 0,
    };

    if role_level >= required_level {
        Ok(())
    } else {
        Err(format!("insufficient permissions: requires {}, you are {}", required, role))
    }
}
```

**安全说明**：角色控制是客户端强制（client-side enforcement）。具有 Git 仓库写权限的用户在理论上可以绕过。这是 v1 的有意简化 — 对于 Termex 的用户场景（信任的团队内部），client-side enforcement 足够。

## 十三、风险与缓解

### 13.1 Git 冲突（中风险）

**风险**: 多人同时修改同一个 server JSON 文件，pull 时 fast-forward 失败。

**缓解策略**:
1. LWW（Last Writer Wins）自动解决 — `updated_at` 更新者覆盖
2. Pull 前先 stash 本地变更，merge 后 unstash + 重新导出
3. 极端情况（仓库 diverge 太远）：force reset to remote + 重新 export 本地 shared

### 13.2 Passphrase 遗失（高风险）

**风险**: 忘记团队密码 → 所有共享凭据无法解密。

**缓解策略**:
1. Admin 可以"重置团队密钥" — 用旧密钥解密后用新密钥重新加密
2. 如果旧密钥也丢失 → 所有成员需要重新配置凭据
3. UI 中提供密码强度提示和安全建议

### 13.3 git2 crate 编译复杂度（低风险）

**风险**: `git2` 依赖 libgit2（C 库），跨平台编译可能遇到问题。

**缓解策略**:
1. 使用 `bundled` feature 自动编译 libgit2
2. Team 功能作为可选 feature gate，不影响基础构建
3. CI 验证三平台 (macOS/Windows/Linux) 编译

### 13.4 大仓库性能（低风险）

**风险**: 数百台服务器 + 长期 Git 历史导致 clone/pull 变慢。

**缓解策略**:
1. 每台 server 一个小 JSON 文件，文件粒度最小化 diff
2. `git2` clone 时可指定 `--depth 1`（shallow clone）
3. 定期建议团队 admin squash 历史

### 13.5 成员泄露（中风险）

**风险**: 前成员保留本地 clone，仍持有团队密钥。

**缓解策略**:
1. Admin 移除成员后，建议更换团队 passphrase
2. Passphrase 更换后，旧密钥无法解密新密文
3. 更换流程：admin 输入旧密码 → 输入新密码 → 重新加密所有文件 → commit + push
4. UI 中在成员移除后主动提示 passphrase 轮换

## 补充设计 A：Passphrase 输入时机与缓存策略

### A.1 生命周期

团队密钥（从 passphrase 派生的 32 字节 AES key）的内存生命周期：

```
┌──────────────────────────────────────────────────────────┐
│                    应用启动                               │
│  ├─ 检查 settings.team_repo_path 是否存在                 │
│  ├─ 如果已加入团队:                                       │
│  │   ├─ 自动尝试从 OS Keychain 读取缓存的 passphrase     │
│  │   ├─ 成功 → 派生团队密钥 → 存入 state.team_key        │
│  │   └─ 失败 → team_key = None（延迟到首次同步时弹框）    │
│  └─ 如果未加入团队: 无操作                                │
│                                                          │
│                    首次同步                               │
│  ├─ team_key == None → 弹出 passphrase 输入对话框         │
│  ├─ 验证: 尝试解密 team.json 中的 _verify 字段            │
│  ├─ 成功 → 派生密钥 → 存入 state.team_key                 │
│  ├─ 用户勾选"记住密码" → 存入 OS Keychain                 │
│  └─ 失败 → 提示密码错误                                   │
│                                                          │
│                    应用退出                               │
│  └─ state.team_key.zeroize() — 内存密钥归零               │
└──────────────────────────────────────────────────────────┘
```

团队 passphrase 缓存复用现有 keychain 单条目设计（参考 CLAUDE.md Keychain Single-Prompt Rule）：

### C.2 重试策略

```rust
const MAX_RETRIES: usize = 3;
const RETRY_DELAY_MS: [u64; 3] = [1000, 3000, 10000]; // 1s, 3s, 10s 递增

async fn sync_with_retry(state: &AppState, credentials: &GitCredentials) -> Result<TeamSyncResult, String> {
    for (attempt, delay) in RETRY_DELAY_MS.iter().enumerate() {
        match team_sync_inner(state, credentials).await {
            Ok(result) => return Ok(result),
            Err(e) if is_network_error(&e) && attempt < MAX_RETRIES - 1 => {
                tokio::time::sleep(Duration::from_millis(*delay)).await;
                continue;
            }
            Err(e) => return Err(e),
        }
    }
    Err("sync failed after retries".to_string())
}
```

## 补充设计 E：SSH Key 共享策略

### E.1 设计决策

**不共享私钥内容**。原因：

1. 私钥文件可能很大（RSA 4096 bit），不适合存入 Git
2. 同一私钥被多人使用违反安全最佳实践（无法追踪谁使用了密钥）
3. 私钥泄露后无法精确吊销单人访问

### E.2 推荐方案

共享服务器如果使用密钥认证，每位成员应配置自己的密钥：

```json
{
  "id": "server-uuid",
  "auth_type": "key",
  "key_path": null,
  "key_hint": "请配置你的 SSH 私钥路径（每位成员使用自己的密钥）",
  "password_enc": null
}
```

导入时：
1. 如果 `auth_type == "key"` 且 `key_path == null` → 提示用户选择本地私钥文件
2. 选择后存入本地 DB，不同步到团队仓库

### E.3 Password 认证的共享

`auth_type == "password"` → `password_enc` 正常加密共享，所有成员使用相同密码。

`auth_type == "agent"` → 无凭据需要共享，每位成员使用自己的 SSH Agent。

### G.2 后端实现

```rust
#[tauri::command]
pub async fn team_rotate_key(
    state: State<'_, AppState>,
    old_passphrase: String,
    new_passphrase: String,
) -> Result<(), String> {
    check_role(&state, "admin")?;

    // 1. 从 old_passphrase 派生旧密钥
    let team_json = read_team_json(&state)?;
    let salt = hex::decode(&team_json.salt).map_err(|e| e.to_string())?;
    let old_key = derive_team_key(&old_passphrase, &salt.try_into().unwrap());

    // 2. 验证旧密码正确
    team_decrypt(&old_key, &team_json.verify)
        .map_err(|_| "current passphrase is incorrect".to_string())?;

    // 3. 生成新 salt + 派生新密钥
    let mut new_salt = [0u8; 16];
    ring::rand::SystemRandom::new().fill(&mut new_salt).unwrap();
    let new_key = derive_team_key(&new_passphrase, &new_salt);

    // 4. 重新加密所有文件
    let repo_path = state.team_repo_path.read().await
        .clone().ok_or("not in a team")?;

    // servers/*.json — 重新加密 password_enc / passphrase_enc
    for entry in std::fs::read_dir(repo_path.join("servers"))? {
        let path = entry?.path();
        if path.extension().map(|e| e == "json").unwrap_or(false) {
            let content = std::fs::read_to_string(&path)?;
            let mut server: serde_json::Value = serde_json::from_str(&content)?;
            re_encrypt_field(&mut server, "password_enc", &old_key, &new_key)?;
            re_encrypt_field(&mut server, "passphrase_enc", &old_key, &new_key)?;
            std::fs::write(&path, serde_json::to_string_pretty(&server)?)?;
        }
    }

    // proxies/proxies.json — 重新加密 username_enc / password_enc
    re_encrypt_proxies(&repo_path, &old_key, &new_key)?;

    // 5. 更新 team.json: 新 salt + 新 _verify
    team_json.salt = hex::encode(new_salt);
    team_json.verify = team_encrypt(&new_key, "TERMEX_TEAM_VERIFY")?;
    write_team_json(&repo_path, &team_json)?;

    // 6. 更新内存中的团队密钥
    *state.team_key.write().await = Some(new_key);

    // 7. Commit + push
    let credentials = load_git_credentials(&state)?;
    let repo = TeamRepo::open(&repo_path)?;
    repo.commit_and_push("rotate team key", &current_username(&state)?, &credentials)?;

    // 8. 更新 keychain 缓存（如果启用了记住密码）
    if let Ok(_) = state.keychain.get("team_passphrase") {
        state.keychain.store("team_passphrase", &new_passphrase)?;
    }

    Ok(())
}

fn re_encrypt_field(
    obj: &mut serde_json::Value,
    field: &str,
    old_key: &[u8; 32],
    new_key: &[u8; 32],
) -> Result<(), String> {
    if let Some(enc) = obj.get(field).and_then(|v| v.as_str()) {
        let plaintext = team_decrypt(old_key, enc)?;
        let new_enc = team_encrypt(new_key, &plaintext)?;
        obj[field] = serde_json::Value::String(new_enc);
    }
    Ok(())
}
```

### G.3 成员侧体验

密钥轮换后，其他成员下次 sync 时：
1. Pull 成功（Git 层面无问题）
2. 尝试用旧密钥解密 → 失败
3. 弹出 Passphrase 对话框 → 提示"团队密码已更新，请输入新密码"
4. 输入新密码 → 验证 `_verify` → 成功 → 更新本地缓存

### H.1 `team_create` — 创建团队

```
用户输入:
  - 团队名称
  - 团队密码（+ 确认）
  - Git 仓库 URL（必须是空仓库或不存在的 URL）
  - Git 认证方式

处理流程:
  1. 验证密码强度（≥8 字符，含大小写+数字）
  2. 生成 16 字节随机 salt
  3. 派生团队密钥
  4. 创建本地临时目录: ~/.termex/team-repo/
  5. git init
  6. 生成 team.json（name, salt, _verify, members=[当前用户 admin], settings）
  7. 生成 .gitignore
  8. 创建 servers/, groups/, snippets/, proxies/ 空目录（含 .gitkeep）
  9. git add -A && git commit -m "init team: {name}"
  10. git remote add origin {repo_url}
  11. git push -u origin main
  12. 如果 push 失败（仓库不存在/非空）→ 回滚，提示用户先创建空仓库
  13. 存储 settings KV
  14. 审计日志
```

### H.2 `team_join` — 加入团队

```
用户输入:
  - Git 仓库 URL
  - 团队密码
  - 用户名
  - Git 认证方式

处理流程:
  1. git clone {repo_url} → ~/.termex/team-repo/
  2. 读取 team.json → 获取 salt
  3. 从 passphrase + salt 派生密钥
  4. 尝试解密 _verify → 验证密码正确性
  5. 如果失败 → 删除 clone → 提示密码错误
  6. 检查 members 中是否已有该 username → 如果有且 device_id 不同 → 更新 device_id
  7. 如果 username 不存在 → 添加到 members（role: member）
  8. 遍历 servers/*.json → 解密 → 导入本地 DB + keychain
  9. 遍历 snippets/*.json → 导入本地 DB
  10. 导入 groups/groups.json → 合并到本地分组
  11. 导入 proxies/proxies.json → 解密 → 导入本地
  12. git commit -m "join: {username}" && push
  13. 存储 settings KV
  14. 审计日志
```
